勘误平台 (Errata Platform) 数据流转架构

从用户提交勘误 → 审核 → 六步发布管线 → 最终写入文件的完整数据链路

核心结论：勘误不直接修改卡牌数据库

  • • 用户提交的勘误以"修改提案"形式存入 errata 表
  • • original_content 和 modified_content 都是 JSON 字符串
  • • 本地卡牌数据库/ .card 文件在整个流程中始终只读
  • • 审核通过仅改变 errata.status，不触发任何文件写入

数据何时真正写入文件？

  • • 唯一写入时机：管理员执行发布管线 Step 6
  • • 写入目标：SCED-downloads/ 中文 TTS JSON 文件
  • • 修改字段：FaceURL、BackURL、CardID、CustomDeck
  • • 但 data_repo_sync 的 git reset --hard 会强制覆盖

数据流转关键节点

  • • .card 文件 → scanner → card_index/local_card_files（索引）
  • • 前端加载 JSON 副本 → 用户编辑 → errata 表（提案）
  • • 审核通过 → 发布管线 → 渲染 → 图床上传 → TTS 导出
  • • TTS 回写 URL → SCED-downloads 中文 JSON（唯一写入）
